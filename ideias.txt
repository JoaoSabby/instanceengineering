Revisao tecnica das ideias de otimizacao para OverUnderSampling.R
=================================================================

Objetivo: maximizar desempenho sem introduzir erros silenciosos, mantendo a API R
publica em formato de pacote e permitindo fallback seguro quando a biblioteca
nativa nao estiver carregada em usos nao convencionais.

0) Conversao para pacote R
-------------------------
Status: implementado.

Implementado:
- Codigo R movido para R/OverUnderSampling.R.
- Criados DESCRIPTION, NAMESPACE e man/OverUnderSampling.Rd.
- A biblioteca C em src/over_under_fast.c agora e compilada pelo mecanismo padrao
  de instalacao de pacotes R e carregada via useDynLib(instanceengineering,
  .registration = TRUE).
- As chamadas .Call usam PACKAGE = "instanceengineering" para resolver os
  simbolos nativos do pacote instalado de forma explicita.

1) Rcpp/RcppArmadillo
---------------------
Status: descartado nesta iteracao.

Motivo:
- A ideia original acelerava a geracao sintetica ADASYN com Armadillo, mas exigia
  Rcpp e RcppArmadillo.
- Para o kernel necessario, nao ha algebra linear complexa; o trabalho e
  essencialmente indexacao column-major, amostragem RNG e interpolacao linear.
- C puro via .Call reduz dependencias e evita conversoes adicionais.

2) C puro via .Call
-------------------
Status: implementado em src/over_under_fast.c.

Implementado:
- OU_ComputeZScoreParamsC: medias e desvios padrao por coluna em duas passagens.
- OU_ApplyZScoreC: aplica e reverte z-score sem sweep/transposicao.
- OU_GenerateSyntheticAdasynC: gera amostras sinteticas com ponteiros diretos,
  matriz column-major e RNG nativo do R.

Cuidados de seguranca:
- Validacao de tipos e dimensoes antes de acessar ponteiros.
- Checagem de indices de vizinhos antes do acesso a matriz da minoria.
- Uso de GetRNGstate/PutRNGstate para preservar reprodutibilidade com set.seed.
- Fallback automatico para R/Rfast caso a biblioteca nativa nao compile ou nao
  carregue.

3) Fortran puro
---------------
Status: descartado nesta iteracao.

Motivo:
- Fortran e muito bom para kernels densos numericos, mas o gargalo principal do
  ADASYN combina RNG, indices 1-based vindos do R, validacao de limites e escrita
  em matriz R.
- A interface .Call em C e mais direta para manipular SEXP, PROTECT e estado do
  RNG do R.
- Usar Fortran exigiria camadas C adicionais ou maior cuidado com interoperacao,
  sem ganho esperado claro frente ao C puro para este padrao de acesso.

4) Rfast e cli
--------------
Status: mantidos como dependencias leves do fluxo.

Implementado:
- OverUnderLoadPackages exige cli e Rfast.
- Quando o caminho C nao esta disponivel, ComputeZScoreParams usa
  Rfast::colmeans/Rfast::colVars e Apply/RevertZScoreScalingMatrix usa
  Rfast::eachrow.
- Erros, avisos e mensagens internas passam pelos wrappers OverUnderAbort,
  OverUnderWarn e OverUnderInform, baseados em cli, com fallback base apenas se
  cli ainda nao estiver disponivel.

5) Transposicao dupla para escala
---------------------------------
Status: descartado.

Motivo:
- t((t(x) - center) / scale) costuma ser melhor que sweep em alguns cenarios,
  mas cria transpostas completas e aumenta pressao de memoria.
- Para bases grandes, memoria e copia extra podem custar mais do que o ganho.
- C puro e Rfast::eachrow resolvem o mesmo problema sem a transposicao explicita.

6) Inferencia parcial de tipos em apenas 1000 linhas
----------------------------------------------------
Status: descartado por seguranca.

Motivo:
- Pode classificar uma coluna como integer/binary com base em amostra inicial e
  errar quando valores decimais aparecem depois.
- A restauracao de tipos e uma etapa de interpretabilidade; erro aqui seria
  silencioso e dificil de depurar.
- A implementacao mantem varredura completa.

7) KNN com FNN
--------------
Status: mantido por compatibilidade, mas nao tratado como backend paralelo.

Motivo:
- FNN::get.knnx ja existia no fluxo e continua simples, estavel e facil de
  instalar via CRAN.
- A verificacao da documentacao nao indica interface de workers/BPPARAM para
  paralelizar consultas no FNN.
- Portanto, FNN continua como backend padrao/compatibilidade, com
  knnAlgorithm = "auto" selecionando "kd_tree" ate 15 colunas e "brute" acima
  disso, mas nao deve ser vendido como solucao paralela.

8) KNN paralelo com BiocNeighbors/BiocParallel
----------------------------------------------
Status: implementado como backend opcional exato por padrao.

Implementado:
- Novo argumento knnBackend = c("auto", "FNN", "BiocNeighbors", "RcppHNSW").
- Novo argumento knnWorkers para configurar workers quando BiocNeighbors ou
  RcppHNSW sao usados.
- Novo argumento biocNeighborAlgorithm = c("auto", "Kmknn", "Vptree",
  "Exhaustive", "Annoy", "Hnsw").
- GetKnnx centraliza chamadas a FNN::get.knnx, BiocNeighbors::queryKNN ou
  RcppHNSW::hnsw_build/RcppHNSW::hnsw_search.
- O backend BiocNeighbors usa BiocParallel::SerialParam, MulticoreParam ou
  SnowParam, conforme knnWorkers e sistema operacional.

Decisao de seguranca:
- biocNeighborAlgorithm = "auto" escolhe apenas metodos exatos por padrao:
  "Kmknn" para baixa dimensao e "Exhaustive" para alta dimensao.
- "Annoy" e "Hnsw" ficam disponiveis, mas sao aproximados e devem ser usados
  apenas quando a aproximacao for aceitavel para o usuario.

9) RcppHNSW
-----------
Status: implementado como backend opcional aproximado, nao como padrao.

E uma boa ideia?
- Sim, quando o gargalo dominante e KNN em bases grandes e pequenas diferencas
  nos vizinhos sao aceitaveis.
- O pacote expoe n_threads, hnsw_build e hnsw_search, o que permite paralelismo
  e busca HNSW rapida via CRAN.
- Nao deve ser o default do pipeline porque HNSW e aproximado: pode alterar os
  vizinhos, a dificuldade ADASYN por linha, as amostras sinteticas e as linhas
  retidas pelo NearMiss.

Implementado:
- knnBackend = "RcppHNSW".
- Parametros hnswM e hnswEf nas funcoes publicas.
- Remocao robusta do proprio ponto em consultas self-KNN, pois metodos
  aproximados podem nao retornar o proprio ponto sempre na primeira posicao.

10) Otimizacoes futuras recomendadas
------------------------------------
- Criar microbenchmarks formais quando R estiver disponivel no ambiente de CI.
- Medir FNN versus BiocNeighbors versus RcppHNSW com diferentes n, p, k e
  workers; o melhor backend pode variar conforme dimensao, tamanho da base,
  tolerancia a aproximacao e custo de paralelizacao.
- Se o projeto virar pacote R completo, mover OverUnderSampling.R para R/ e usar
  NAMESPACE/useDynLib para carregar src/over_under_fast.c sem compilacao em
  tempdir.
